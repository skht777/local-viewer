# 基本行動指針

## 言語・コミュニケーション
- 日本語で応答する
- 簡潔に要点だけ伝える。冗長な説明は不要
- コード内のコメントは日本語で記述

## 品質原則
- Clean Code を常に意識する
- セキュリティリスクのあるコード（SQLインジェクション、XSS、パストラバーサル等）は絶対に提案しない
- やむを得ない場合は警告を表示し、安全な代替案を提示する
- 複雑なロジックはまず擬似コードやコメントで設計してから実装する

## 作業スタイル
- Tidy First: 機能変更やバグ修正の前に、まずコードを整理する
- リファクタリング（構造変更）と機能追加（振る舞い変更）は別のコミットで行う
- 既存のコード・パターン・ユーティリティを最大限再利用する。不要な新規作成をしない
- ファイルは最大500行（空行・コメント除外）を目安に分割を検討する

## プロセス
- 実装前に必ず要件を確認し、不明点があれば質問すること。
- 複雑なロジックは、擬似コードやステップごとの解説を行ってから実装コードを提示すること。
- 重要な変更（新機能、アーキテクチャ変更、基盤整備等）の実装計画は `docs/plan-<名前>.md` に保存する。実装完了後も設計意図の記録として残す。
