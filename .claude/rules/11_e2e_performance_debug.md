# E2E パフォーマンス・デバッグ規約

## パフォーマンス設定

### 現在の設定 (playwright.config.ts)

- webServer: バックエンド(8001) + フロントエンド(5174) を自動起動
- ブラウザ: Chromium のみ
- リトライ: 1回
- トレース: `on-first-retry`
- スクリーンショット: `only-on-failure`

### 実行時間の目安

- スモークテスト: 5秒以内
- ナビゲーション/CRUD フロー: 15秒以内
- 複雑なワークフロー（アーカイブ展開等）: 30秒以内
- 上記を超えるテストは分割を検討

### CI 環境

- `workers: 2`（vCPU数に合わせて調整）
- `~/.cache/ms-playwright` をキャッシュ
- Chromium のみインストール: `npx playwright install --with-deps chromium`

## テスト失敗時の対応フロー

1. エラーメッセージとスタックトレースを確認
2. Playwright MCP の `browser_snapshot` でページの現在状態を確認
3. 以下の原因を順にチェック:
   - **タイミング問題** → 適切な待機処理を追加（`waitForTimeout` 以外）
   - **セレクタ変更** → `data-testid` ベースのロケーターに更新
   - **アプリケーション側の変更** → ヘルパー関数または PO クラスを更新
   - **テストデータの問題** → フィクスチャを修正
4. 修正後にテストを再実行して確認
5. 3回修正しても解決しない場合は、問題を報告して人間の判断を仰ぐ

## Flaky テスト対策

- リトライで通過するテストは根本原因を調査する
- 主な原因と対策:
  - **レースコンディション** → `waitForResponse()` / `waitForSelector()` を追加
  - **アニメーション** → `animations: 'disabled'` を設定に追加
  - **ネットワーク遅延** → タイムアウト調整ではなくレスポンス待機を使用
  - **サムネイル読み込み** → 画像の `load` イベントを待機

## ログ・レポート

- ローカル: `html` + `list` レポーター
- CI: `html` レポーター + `upload-artifact`
- 失敗時のトレースは `trace: 'on-first-retry'` で自動保存
- レポート表示: `cd e2e && npx playwright show-report`

## コードレビュー時のチェックポイント

- [ ] テスト名が何を検証するか明確か
- [ ] `waitForTimeout` が使われていないか
- [ ] ハードコード URL が含まれていないか
- [ ] アサーションがビジネスロジックを検証しているか
- [ ] テスト間の依存関係がないか
- [ ] `test.only()` や `console.log` が残っていないか
- [ ] `force: true` が使われていないか（使用時は理由をコメント）
