# プラン策定規約

## プランが必要な変更
- 新機能の追加（複数ファイルに影響）
- アーキテクチャ変更（依存方向変更、新レイヤー追加）
- 基盤変更（テスト基盤、ビルド設定、Docker構成）

プラン不要: 単一ファイルのバグ修正、設定値変更、コメント/ドキュメントのみ、lint修正

## 重要度分類

変更の影響範囲に応じて重要度を判定し、プランの冒頭に明記する:

| 重要度 | 基準 | レビュー要件 |
|--------|------|------------|
| S | アーキテクチャ変更、セキュリティモデル変更、新外部依存 | codex review 必須 |
| A | 新機能、複数ファイル変更（5+）、API breaking change | codex review 推奨 |
| B | バグ修正、単一〜少数ファイル、テスト追加 | セルフレビューのみ |
| C | 設定変更、コメント修正、lint修正 | プランモード不要 |

## プランの構造（必須セクション）

1. **Context**: 背景・目的・前提条件・適用ルールの要約
2. **Approach**: 実装方針・代替案と選択理由
3. **Files**: 変更対象一覧（新規/変更/削除の区分付き）
4. **Verification**: テスト計画・lint/型チェック・手動確認項目

任意セクション:
- **申し送り**: 次フェーズへの注意事項
- **コミット計画**: Tidy First に基づく分割案

### 起動時処理を追加するプランの Verification 必須項目
`build_app` など起動時の同期処理を追加するプランでは、Verification セクションに以下を必ず含める:
- 起動時間の受入基準（N entries で < X ms のように定量）
- RSS/メモリ増分の受入基準（1M entries で < Y MB のように定量）
- 閾値超過時の**縮退モード**定義と観測手段（`/api/health` の diagnostics フィールド、構造化ログ等）

## プラン策定前の調査

1. `docs/spec-*.md`（現行仕様書）と `docs/` 配下の関連ファイル（`docs/plan-*.md` 等）から現在の状況・経緯・仕様を把握する
2. 類似機能の既存実装パターン（Grep/Glob）
3. 変更対象ファイルの現在の状態（Read）
4. 依存方向（02_architecture.md に合致するか）
5. セキュリティ影響（ファイルアクセス・ユーザー入力を扱うか）
6. 設計判断に迷う場合は、当該概念のベストプラクティスやワークアラウンドをWeb検索する

## セルフレビュー（提示前に最低1回）

1. 既存パターンとの整合性
2. 依存方向の遵守（02_architecture.md）
3. セキュリティ影響の考慮（09_security.md）
4. 過剰設計の回避（不要な抽象化がないか）
5. コンテキストウィンドウ効率（不要な情報を含めていないか）

## 不明点の解決

- 設計判断に迷いが生じた場合、推測で進めずWeb検索で当該概念のベストプラクティスやワークアラウンドを確認する
- 検索結果をプランの Approach セクションに根拠として記載する

## codex レビュー連携

- 重要度 S: plan-reviewer エージェントを起動し、codex exec でプランをレビュー
- 重要度 A: プラン完成後にユーザーに codex レビューを提案
- plan-reviewer はレビュー結果とともに「更新された適用ルール」も出力する。反映時に適用ルールも同時に更新すること
